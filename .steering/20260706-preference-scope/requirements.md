# 要求内容：F8 学習済み好みの適用スコープ導入

## 1. 背景・問題

F8 フィードバック学習で抽出された `learned_preferences` は `{text, created_at}` のフラットなリストとして保存され、
`orchestrator.py` の `profile_text` 構築時に**全件**が「以下の好みを成果物の生成方針に必ず反映してください」という
無条件命令付きで、トピック解析・ワークフロー設計・全生成ステップの**全プロンプト**に注入される。

このため、例えば IaC コードへのフィードバック由来の好み（「Terraformコードはmodule分割する」）が
時事トピックの調査レポート生成にも「必ず反映」で混入し、**トピックと無関係な好みが成果物の品質を下げる**障害が発生している。

抽出プロンプトは元スレッドの topic / category を参照しているにもかかわらず、保存時にスコープ情報を捨てていることが根本原因。

## 2. 変更・追加する機能

学習済み好みに**適用スコープ**（カテゴリ × 成果物区分）を付与して保存し、
パイプラインの各段階で**コードによる決定的フィルタ**を行って関連する好みだけをプロンプトに注入する。

グリリングセッション（2026-07-06）で確定した設計判断：

| # | 決定 |
|---|---|
| 1 | 関連性判定は**型（データ）層**を主軸に置く。プロンプト層の文言緩和（「必ず」の使い分け）を併用。実行時 LLM 判定（パイプライン層のみ案）は不採用 |
| 2 | スコープは `{categories: [], deliverables: []}` の 2 次元。**両方空 = 汎用**（全プロンプト注入対象） |
| 3 | **段階的絞り込み**: ①トピック解析 = 汎用のみ / ②ワークフロー設計 = 汎用 + カテゴリ一致（成果物スコープ付きはラベル付きで提示し「該当成果物を選ぶ場合に考慮」と緩和）/ ③生成ステップ = 完全フィルタ（「必ず反映」を維持） |
| 4 | スコープ付与は抽出 LLM。**非対称フォールバック**: 意図的な空リスト = 汎用 / enum 検証失敗による空化 = 元実行のスコープに縮退（迷ったら狭く。過剰適用より過少適用の方が害が小さい） |
| 5 | 既存好みは**一括再分類スクリプト**で移行。書き戻し前に分類結果を目視確認 |
| 6 | 置換判定（replaces_index）は LLM 継続、既存好み一覧にスコープを明示提示。保存上限 **10 → 20 件**、FIFO 維持 |
| 7 | Slack 記録通知と /profile 画面にスコープラベル表示。**訂正チャネルは F8 フィードバック自体に一本化**（編集 UI は作らない） |
| 8 | 好みスコープの成果物区分は **6 値**（コード / 調査レポート / アーキテクチャ設計書 / 比較表 / 料金見積もり / 手順書）。「コード」はフィルタ時に `{iac_code, program_code}` へ展開。ワークフロー内部の `deliverable_type` 7 種は**不変** |

## 3. ユーザーストーリー

- ユーザーとして、コードの書き方についてフィードバックしたら、その好みは**コード成果物の生成時だけ**に反映されてほしい。時事レポートの文体には影響してほしくない。
- ユーザーとして、「結論を先に簡潔に」のような汎用的な好みは、**すべての成果物**に引き続き反映されてほしい。
- ユーザーとして、フィードバックが**どの範囲に適用されるものとして記録されたか**を Slack の応答と /profile 画面で確認したい。
- ユーザーとして、記録されたスコープが意図と違ったら、**同じスレッドにもう一度フィードバックする**だけで訂正したい。

## 4. 受け入れ条件

1. **ミスマッチ解消（本障害の修正）**: `deliverables: ["code"]` スコープの好みを持つ状態で時事トピック（成果物 research_report のみ）を実行したとき、その好みが ① と ③（research_report 生成プロンプト）に注入されない。② では決定 3 のとおり「該当する成果物を選ぶ場合に考慮」の緩和文言付き小節にのみ現れうる（成果物選択の材料としての提示であり、生成内容への「必ず反映」命令ではない）
2. **汎用好みの維持**: スコープが両方空の好みは ①②③ すべてのプロンプトに注入される
3. **段階的絞り込み**: カテゴリ「技術」スコープの好みは、技術トピック実行時の ② と ③ に注入され、①には注入されない。成果物「コード」スコープの好みは ② でラベル+緩和文言付き、③ では iac_code / program_code の生成ステップにのみ「必ず反映」で注入される
4. **抽出とバリデーション**: フィードバック処理後、保存された好みに `scope` が付与されている。LLM が enum 外の値を出した場合はその値だけ破棄され、破棄の結果次元が空になった場合は元実行の category / 成果物区分に縮退する。LLM が意図的に空を出した場合は汎用として保存される
5. **可視化**: Slack 記録完了通知の各好みに `[汎用]` `[技術・コード]` 等のスコープラベルが付く。/profile 画面の学習履歴にスコープバッジが表示される
6. **移行**: 既存の学習済み好み全件が再分類スクリプトで scope 付きに移行されている（dry-run 出力を目視確認してから書き戻し）
7. **上限**: 好みは 21 件目の追加で最古の 1 件が削除される（上限 20）
8. **後方互換**: `scope` フィールド欠損の好みレコードに遭遇してもエラーにならず汎用として扱われる（防御的読み取り）

## 5. 制約事項

- ワークフロー内部の成果物タイプ enum（`deliverable_type` 7 種）およびオーケストレーターの成果物選択ロジックは変更しない
- 新規 AWS リソース・IAM 変更なし（`template.yaml` 変更なし）
- スコープ訂正のための新規 UI / Slack Modal は作らない
- ダッシュボードのフィードバック集計（`get_feedback_aggregation` / FeedbackAnalysis.tsx）は変更しない（件数集計にスコープは影響しない）
- CLAUDE.md §4 に従い、実装コミット後は Codex レビューゲートを通過し、ユーザー明示承認後にのみ `sam build` / `sam deploy` を実行する
