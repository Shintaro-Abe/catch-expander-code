# requirements.md — Slack Modal によるユーザープロファイル登録 (F6)

## 背景

`docs/product-requirements.md` の F6（ユーザープロファイル）は **部分実装** 状態:
- ✅ `UserProfilesTable` (DynamoDB) と `get_user_profile` / `put_user_profile` API 実装済み
- ✅ orchestrator が `learned_preferences` 含むプロファイルを参照（production 反映済み）
- ✅ F8 フィードバック学習で `learned_preferences` が自動累積
- ❌ `@CatchExpander profile` での対話式登録 UX **未実装**
- ❌ スキーマバリデーション **未実装**

`docs/functional-design.md` §5.2 のシーケンス図は設計のみで、本作業でこれを実装する。

## ユーザーストーリー

- **As a** Catch Expander ユーザー
- **I want** Slack 上で `@CatchExpander profile` と入力するだけで自分のロール・専門領域などを登録・更新できる UI を使いたい
- **So that** DynamoDB 直接書き込みや CLI 操作不要で、orchestrator のプロンプトに自分の文脈を反映できる

## 受け入れ条件

### AC-1: 起動コマンドの検出
- ユーザーが `@CatchExpander profile` と Slack で投稿すると、Bot は **「プロファイルを開く」ボタン付きメッセージ** を同チャンネルに返す
- 既存の `@CatchExpander <トピック>` ルーティング（オーケストレーター起動）と衝突しない（"profile" は予約語）

### AC-2: Modal の表示
- ボタン押下 (`block_actions` イベント) で Slack の **Modal が開く**
- Modal タイトル: "プロファイル登録"
- Modal の最上部に **ヘルプ section block** を 1 行表示:
  > 💡 具体例を交えて 100〜300 字で書くと、出力の精度が上がります。詳しい書き方は各フィールドの placeholder を参考にしてください。
- 入力フィールド 6 つ（IT・非 IT 問わない汎用設計、各 `plain_text_input` に **ベストプラクティスに沿った placeholder** を設定）:

  1. **役割・職業** (`role`) - text input
     - placeholder: `クラウドエンジニア（AWS 中心、SRE 担当） / マーケター（B2B SaaS） / 大学院生（環境工学）`
  2. **関心分野** (`interests`) - multi-line text input
     - placeholder: `AI（LLM のビジネス活用）、料理（時短レシピ）、投資（インデックス中心）など、深さや方向性を括弧で添えて`
  3. **専門・得意領域** (`expertise`) - multi-line text input
     - placeholder: `インフラ設計（5 年）、Python・SQL は実務レベル、統計は基礎のみ — 何が「できる/できない」かを書くと精度が上がります`
  4. **学習の目的** (`learning_goals`) - multi-line text input
     - placeholder: `副業として Web 制作で案件獲得したい。HTML/CSS は理解しているが、営業ノウハウが弱い — Why と現在地を明示`
  5. **背景・状況** (`background`) - text input
     - placeholder: `30 代後半、子育て中、SaaS 企業のインフラチーム所属、転職活動準備中`
  6. **受け取り方の好み** (`output_preferences`) - multi-line text input
     - placeholder: `箇条書き重視、実例を 3 つ必ず添える、専門用語は注釈付き、英語ソース引用 OK、政治・宗教の話題は避ける`

- 既存プロファイルがあれば各フィールドに **既存値を初期表示**（編集モード）。空のフィールドのみ placeholder が表示される。

- フィールド設計の意図:
  - すべて自由記述・任意で、IT 業界限定語や ENUM 制約を排除（時事・趣味・教養を含む幅広いトピックに対応）
  - 「受け取り方の好み」は orchestrator が出力スタイル・深さを調整する材料となる
  - 既存の `learned_preferences`（F8 で自動累積）と相補的: 本フィールドは明示的な自己申告、F8 はフィードバックからの学習

### AC-2.5: プロファイル記載のベストプラクティス（LLM 出力品質への寄与）

本作業では、プロファイルが orchestrator のプロンプトに挿入された際に **LLM 出力の精度を上げる** ことを最終目的とする。
そのため、Modal の placeholder とヘルプ section は以下のベストプラクティスを反映する設計とする。

1. **具体性 > 抽象性**: 「IT」「ビジネス」など抽象語ではなく、「クラウドアーキテクチャ（AWS 中心）」のように深さを示す
2. **単語列挙より文脈付き**: タグ列挙ではなく `AI（特に LLM 活用）` のように括弧で方向性を添える
3. **Why と現在地の明示**: 「副業準備」ではなく「副業として Web 制作の案件獲得を目指す。HTML/CSS は理解しているが営業ノウハウが弱い」
4. **ネガティブ情報も有用**: 「JavaScript には深入りしない」「政治・宗教の話題は避けたい」など除外条件を歓迎
5. **役割と背景は別軸**: 「役割」= 何をしている人か / 「背景」= どんな状況・ライフステージか の区別を保つ
6. **「受け取り方の好み」は output 制御に直結**: 「箇条書き重視」「実例 3 つ必須」など出力スタイルを直接制御できる項目として強調
7. **100〜300 字を目安**: ノイズが増えると判定が散漫になるため、凝縮された記述を推奨
8. **更新前提**: 関心は変わるため、ライフイベント時の再登録を想定（再登録 UX は AC-2 の編集モードで充足）

これら原則は **placeholder の文言** と **ヘルプ section の 1 行** で示唆し、強制ではなく推奨に留める（自由度を保持）。

### AC-3: Submit 処理
- 「保存」ボタン (`view_submission` イベント) 押下で:
  - 6 フィールドのうち **値が入っているフィールドのみ** DynamoDB `UserProfilesTable` に書き込み
  - **空欄のフィールドは Item から該当キーを削除（REMOVE）** する → "空欄保存 = そのフィールドを未登録に戻す" 挙動（B 方式）
  - `user_id` は Slack の user.id をそのまま使用
  - `learned_preferences` フィールド（既存）は **上書き / 削除しない**（マージで保持）
  - 実装: `put_user_profile()` を「空文字キーを除外した dict を UpdateItem `SET` + 空欄キーを `REMOVE`」で実装、または get → merge → put の経路で空文字キーを dict から除外
- 成功時、Bot は元の起動チャンネル（または DM）に **確認メッセージ** を投稿
  - 「✅ プロファイルを保存しました」+ 保存された値のサマリ（空欄にしたフィールドは「削除しました」と明示）

### AC-3.5: フィールド削除挙動（B 方式の明示）
- **個別フィールドの削除**: Modal で対象フィールドを空欄にして「保存」 → DynamoDB Item から該当キーが REMOVE される
- **削除後のデータ状態**: DynamoDB Console で見ると、削除されたフィールドキーは **存在しない**（空文字も残らない）
- **orchestrator 側の影響**: プロンプト構築時に「キー存在チェック」のみで該当軸をスキップできる（空挿入の防止）
- **再度 Modal を開いた時**: 削除されたフィールドは初期値が空、placeholder が表示される（未登録と同じ見た目）
- **プロファイル全体（Item ごと）の削除はスコープ外**（DynamoDB Console 手動削除で代替）

### AC-4: バリデーション
- 全 6 フィールドは **任意（空欄可）**（最初の登録で全項目埋まらないケースがあるため）
- 各フィールド最大文字数: **500 文字**（`view_submission` の validation 経路で Slack ネイティブの error 表示）
- 文字列はそのまま保存（カンマ区切り等のパースは行わない、後の orchestrator 側プロンプト挿入で文字列として扱う）
- 空文字（trim 後 0 文字）は AC-3 / AC-3.5 に従い**フィールド削除**として扱う（バリデーションエラーにはしない）

### AC-5: 既存機能の非干渉
- `@CatchExpander <profile 以外>` の通常トピック投入が変わらず動作する
- F8 フィードバック（スレッド返信）と F9 履歴コマンド (`履歴` / `history`) が変わらず動作する
- `view_submission` 経路は既存の Event API JSON path とは分岐され、相互に影響しない

### AC-6: テスト
- `tests/unit/trigger/test_app.py` に新規ケース追加:
  - "profile" 検出でボタンメッセージを返す
  - `block_actions` payload で `views.open` API が呼ばれる
  - `view_submission` payload で UserProfilesTable に書き込まれる
  - `view_submission` でバリデーションエラーが発生（500 文字超）
- 既存 trigger ユニットテストが全 pass

### AC-7: デプロイ後の動作確認
- sam deploy 後、実際の Slack ワークスペースで:
  - `@CatchExpander profile` 投稿 → ボタンメッセージ表示
  - ボタン押下 → Modal 表示（既存値があれば初期値入り）
  - 6 フィールド入力して保存 → 確認メッセージ
  - DynamoDB Console で UserProfilesTable に書き込まれたことを確認

## フィールド選定根拠（5W1H 直交フレーム）

6 つのフィールドは **5W1H フレーム** に対応する直交した次元として選定されている。
それぞれが互いに代替不能で、削るとプロファイルの表現力が下がり、追加すると冗長化する。
現在の 6 軸が **最小十分集合** である。

| 軸 | フィールド | LLM への効果 |
|---|---|---|
| WHO | 役割・職業 | 読者像の決定（語り口・専門用語許容度）|
| WHAT | 関心分野 | トピック範囲の絞り込み |
| HOW WELL | 専門・得意領域 | 説明の深さ・前提知識レベル |
| WHY | 学習の目的 | ギャップを埋める方向ベクトル |
| WHERE/WHEN | 背景・状況 | 時間制約・ライフステージ文脈 |
| HOW | 受け取り方の好み | 出力フォーマット直接制御 |

### 1. 役割・職業 (`role`) — WHO（あなたは誰か）

- **選定理由**: LLM が「読者像」を最初にイメージするための**基本アンカー**。これがないと LLM は「一般人向け」の平均的な内容を出力しがち。
- **LLM 出力への影響**:
  - 同じ「AI」トピックでも、エンジニア向けは API・実装、マーケター向けは活用事例・ROI、学生向けは概念と将来性、と論点が変わる
  - 専門用語の許容度、参照すべき業界の文脈（B2B / 教育 / メディア など）が決まる
- **代替不能性**: 関心分野や専門領域だけでは「あなたが誰として読むか」が定まらない。

### 2. 関心分野 (`interests`) — WHAT（何を知りたいか）

- **選定理由**: トピックの **WHAT を絞る** 軸。役割と独立して持つことが重要。役割が「マーケター」でも、個人として AI や哲学に興味があるケースは普通にある。
- **LLM 出力への影響**:
  - トピック選定、関連分野の優先順位、深掘り方向の判断材料
  - 「関心: AI（LLM のビジネス活用）、投資（インデックス）」のように方向性を絞ると、LLM は外周のノイズを削れる
- **代替不能性**: 役割 = 仕事、関心 = 個人としての興味、と次元が違う。

### 3. 専門・得意領域 (`expertise`) — HOW WELL（どこまで分かるか）

- **選定理由**: 知識レベル・前提を判断する軸。**説明の深さ**を決定する。
- **LLM 出力への影響**:
  - 「LLM の基礎から説明」か「実装パターンに踏み込む」かの分岐
  - 「何ができて、何ができないか」を書くことで LLM がギャップを埋める出力を選びやすい
  - これがないと LLM は無難な「中級者向け」に丸める傾向
- **代替不能性**: 役割（「エンジニア」）と専門レベル（「5 年実務」or「学習始めて 3 ヶ月」）は別軸。

### 4. 学習の目的 (`learning_goals`) — WHY（なぜ知りたいか）

- **選定理由**: **WHY を明示**するための軸。同じ役割・関心でも目的次第で出力の方向性が変わる。
- **LLM 出力への影響**:
  - 「業務直結のスキルアップ」vs「教養・趣味」で語り口が変わる
  - 「副業準備」「受験対策」「資格取得」「キャリアチェンジ」など、時系列ゴールに沿った提案ができる
  - 「ギャップを埋める情報」を LLM が選びやすくなる（現在地 → 目的地のベクトル提供）
- **代替不能性**: 関心が「AI」でも、目的が「研究」と「副業」では情報の選び方が全く違う。

### 5. 背景・状況 (`background`) — WHERE/WHEN（どんな状況にいるか）

- **選定理由**: **時間的・環境的制約**を伝える軸。役割（What）とは独立した次元。
- **LLM 出力への影響**:
  - 「育児中」→ 短時間で読める / 実践的 / すぐ役立つ、を暗黙に要求
  - 「転職活動中」→ 採用市場の動向 / 自己 PR の文脈、を考慮できる
  - 「学生」→ 学習時間が確保しやすい / 長期スパンの議論 OK
  - 業界・ライフステージのコンテキストで LLM が「あなたに今届くべき情報」を選別できる
- **代替不能性**: 役割（職業）と背景（ライフステージ・所属）は混同されがちだが、出力への影響が違う。

### 6. 受け取り方の好み (`output_preferences`) — HOW（どう受け取りたいか）

- **選定理由**: 唯一 **LLM の出力スタイル** を直接制御する軸。1〜5 がプロファイル「データ」なら、これは「フォーマット指令」。
- **LLM 出力への影響**:
  - 「箇条書き重視」→ 構造化された出力
  - 「実例 3 つ必須」→ 抽象論で終わらない
  - 「専門用語は注釈付き」→ 専門領域が高くても説明的に
  - 「英語ソース引用 OK」→ 一次情報の幅が広がる
  - 「政治・宗教は避ける」→ 出力フィルタ
  - このフィールドは orchestrator が**最も直接的にプロンプトに反映**する
- **代替不能性**: データ系（What / Why / Background）と形式系（How）は性質が全く違うため必須分離。

### 設計判断のメタ説明

- **「最小十分集合」原則**: 6 軸より少ないと LLM が補えない次元が発生（例: WHO を削ると読者像が不定、HOW を削ると形式制御不能）。多くするとフィールド数の増加で入力負担とノイズが増す。
- **直交性**: 6 軸は意味的に重複しない。例えば「役割」(WHO) と「専門」(HOW WELL) は混同されやすいが、職業と知識レベルは別の次元。明示的に分離することで、LLM のプロンプト挿入時に意味の混乱を避けられる。
- **拡張可能性**: 将来「**避けたい話題** (`excluded_topics`)」「**直近のイベント** (`recent_events`)」など追加候補はあるが、現状は `output_preferences` と `background` に含めて運用可能と判断（過剰設計を避ける）。

## 制約事項

- **Slack app 設定の手動作業必須**: Slack app の「Interactivity & Shortcuts」を有効化し、Request URL に既存の `/slack/events` を設定する必要がある（ユーザー作業）
- **interactive payload の Content-Type**: `application/x-www-form-urlencoded` で `payload=<URL encoded JSON>` 形式。既存の `json.loads(body)` 経路と分岐が必要
- **コストインパクト ほぼゼロ**: DynamoDB on-demand への書き込みは年に数回想定（プロファイル更新頻度）。月 $0.001 未満
- **IAM policy 拡張**: `TriggerFunction` の policy に `UserProfilesTable` への `dynamodb:GetItem` / `dynamodb:PutItem` を追加

## スコープ外

- **プロファイル全体（Item ごと）の削除 UX** — 個別フィールドの削除は AC-3.5 で実現するが、Item を丸ごと削除する UX は提供しない。必要なら DynamoDB Console から手動削除
- **プロファイル閲覧 UX**（ダッシュボード可視化 + Slack 専用閲覧コマンド） — **次の別 steering で実装予定**。本 steering では登録/編集 Modal までで完結する。
  - 補足: Slack 上では `@CatchExpander profile` を再実行すれば編集 Modal に**現在値が初期表示**されるため、実質的な閲覧手段としても機能する（編集を意図せず開いてキャンセルすれば閲覧のみで完了）
  - ダッシュボードでの可視化（新規 API + frontend ルート）は規模が大きいため別 steering で集中
- learned_preferences の表示 / リセット
- 複数ワークスペース対応
- スキーマバリデーション（ENUM 制約等、自由記述で許容）
- Slack Slash Command (`/profile`) 形式（app_mention 単一経路に統一）
