# 要求定義書：F8 フィードバック学習

## 1. 概要

### 背景と解決する課題

現在のシステムはトピックを受け取って成果物を生成するが、生成後に何も学習しない。
ユーザーが毎回同じ不満（「コードが短い」「日本語が硬い」「クラウドがAWSに偏っている」など）を感じても
プロファイルは変わらず、次の成果物も同様の傾向になる。

この機能は「成果物の完了通知スレッドへの自由テキスト投稿」をフィードバックとして自動検出し、
Claude が構造化した好みを抽出してユーザープロファイルに蓄積する。
次回のトピック処理時にオーケストレーターがその好みを参照することで、
ユーザーが何も変えなくても出力品質が使い込むほど向上していく仕組みを作る。

### 処理フロー概要

```
[ユーザー] Slackのスレッドにフィードバックを投稿
      ↓
[Lambda] thread_tsで完了済み実行を特定 → ECSタスク起動（TASK_TYPE=feedback）
      ↓
[ECS] Claudeがフィードバックテキストを解析 → preferences を抽出
      ↓
[DynamoDB] user-profiles の learned_preferences を更新（最大10件）
      ↓
[Slack] 記録された好みの一覧を通知
      ↓
[次回トピック処理] オーケストレーターが learned_preferences を読んでプロンプトに反映
```

---

## 2. ユーザーストーリー

### US-F8-1: フィードバックの送信
**As a** ユーザー  
**I want to** 成果物の完了通知スレッドで感想・改善要望を自由形式で書く  
**So that** 特別な操作や形式を覚えることなく自分の好みをシステムに伝えられる

**例：**
- 「コードが参考になりました。次回はTerraformのmodule分割まで含めてほしい」
- 「説明が長すぎる。要点だけ箇条書きにしてほしい」
- 「良かった」（具体性がない場合でも受け付ける）

### US-F8-2: フィードバックの反映確認
**As a** ユーザー  
**I want to** フィードバックが受け取られ、どんな好みとして記録されたかを同じスレッドで知りたい  
**So that** 記録内容が意図と一致しているか確認でき、誤解があればすぐに追加フィードバックで修正できる

### US-F8-3: 累積的な品質改善
**As a** ユーザー  
**I want to** 複数回フィードバックを繰り返すことで、好みのリストが充実していく  
**So that** 使い込むほど自分向けにカスタマイズされた成果物が得られる

### US-F8-4: 既存プロファイル保護
**As a** ユーザー  
**I want to** フィードバックがロールや技術スタックなど自分で登録したプロファイルを上書きしない  
**So that** 意図しないプロファイル変更によって成果物の品質が下がることを防げる

## 3. 受け入れ条件

### AC-F8-1: フィードバック検出（Lambda）

| # | 条件 | 期待動作 |
|---|------|---------|
| 1-1 | メッセージの `thread_ts` が `ts` と異なる（スレッド返信） かつ DynamoDB で `slack_thread_ts == thread_ts AND user_id == uid AND status == "completed"` の実行レコードが見つかる | フィードバックとして処理（ECS起動） |
| 1-2 | メッセージの `thread_ts` がないまたは `ts` と同じ（スレッドなし） | 新規トピックとして既存フローを実行 |
| 1-3 | スレッド返信だが対応する実行レコードが見つからない | 新規トピックとして既存フローを実行（フォールバック） |
| 1-4 | スレッド返信で実行レコードが見つかるが `status != "completed"` | 無視（HTTP 200 を返して終了） |
| 1-5 | `bot_id` が付いたメッセージ | 無視（既存挙動と同じ） |
| 1-6 | `x-slack-retry-num` ヘッダーがある | 無視（既存挙動と同じ） |

**DynamoDB 検索実装方針：**  
`workflow-executions` テーブルを `FilterExpression` でスキャンする（`slack_thread_ts` および `user_id` でフィルタ）。
現状は個人利用スケールのため許容する。スキャンコストが問題になる場合は
`workflow-executions` に `slack_thread_ts` をパーティションキーとする GSI を追加する。

### AC-F8-2: Lambda のフィードバック処理

- [ ] フィードバック検出後、Slack の該当スレッドに即時 ACK メッセージを投稿する
  - 投稿テキスト：`📝 フィードバックを受け取りました。プロファイルに反映中...`
- [ ] ECS RunTask を以下の環境変数で起動する

  | 環境変数 | 値 |
  |---------|-----|
  | `TASK_TYPE` | `"feedback"` |
  | `USER_ID` | Slack ユーザー ID |
  | `FEEDBACK_TEXT` | ユーザーが投稿したテキスト（メンション除去後） |
  | `EXECUTION_ID` | 対応する実行レコードの `execution_id` |
  | `SLACK_CHANNEL` | Slack チャンネル ID |
  | `SLACK_THREAD_TS` | 元スレッドの `thread_ts` |

- [ ] ECS 起動後、Lambda は即座に HTTP 200 を返す（フィードバック処理の完了を待たない）

### AC-F8-3: ECS フィードバック処理（main.py）

- [ ] 環境変数 `TASK_TYPE` が `"feedback"` のとき、`orchestrator.run()` の代わりに `feedback_processor.process()` を呼び出す
- [ ] `TASK_TYPE` が未設定または `"topic"` のときは既存フローを維持する

### AC-F8-4: フィードバック解析（feedback_processor.py）

#### 4-1. 前処理
- [ ] 環境変数から `USER_ID`, `FEEDBACK_TEXT`, `EXECUTION_ID`, `SLACK_CHANNEL`, `SLACK_THREAD_TS` を取得する
- [ ] DynamoDB から実行レコード（`execution_id`）を取得し `topic` と `category` を読み込む
  - 実行レコードが見つからない場合はコンテキストなし（`topic="不明"`, `category="不明"`）で処理を続行する
- [ ] DynamoDB からユーザープロファイル（`user_id`）を取得し、既存の `learned_preferences` を読み込む
  - プロファイルが存在しない場合は `learned_preferences` を空リストとして扱う

#### 4-2. Claude による好みの抽出
- [ ] `call_claude()` を使って Claude にフィードバックを解析させ、好み（preferences）を抽出する
- [ ] Claude に渡す情報：
  - 元トピック名とカテゴリ
  - 既存の `learned_preferences` テキスト一覧（重複・矛盾の判断に使用）
  - フィードバックテキスト
- [ ] Claude は以下の JSON を返す：
  ```json
  {
    "preferences": [
      {
        "text": "Terraformコードはmoduleを分割してディレクトリ構造で管理する",
        "replaces_index": null
      }
    ]
  }
  ```
  - `text`：次回の成果物生成プロンプトにそのまま渡せる形式の文章（命令形・具体的）
  - `replaces_index`：既存 `learned_preferences` の何番目を置き換えるか（0始まり）。なければ `null`
  - `preferences` が空配列の場合は「具体的な好みが抽出できなかった」ケースとして扱う
- [ ] 1回のフィードバックから最大3件の preference を抽出する（それ以上は切り捨て）

#### 4-3. マージルール
- [ ] `replaces_index` が `null` でない場合、既存リストの該当インデックスを新しい preference で置き換える
- [ ] `replaces_index` が `null` の場合、既存リストの末尾に追加する
- [ ] マージ後のリストが10件を超えた場合、先頭から超過分を削除する（最古のものが削除される）
- [ ] マージ後の各 preference オブジェクトのフォーマット：
  ```json
  {
    "text": "Terraformコードはmoduleを分割してディレクトリ構造で管理する",
    "created_at": "2026-04-12T12:34:56.789Z"
  }
  ```
  - `replaces_index` で置き換えた場合も `created_at` は新しい時刻で上書きする

### AC-F8-5: ユーザープロファイル更新

- [ ] `learned_preferences` フィールドのみを更新する（`role`, `clouds`, `tech_stack` 等のユーザー登録フィールドは一切変更しない）
- [ ] `updated_at` を現在時刻で更新する
- [ ] プロファイルレコードが存在しない場合、`user_id` と `learned_preferences` のみを持つレコードを新規作成する

### AC-F8-6: Slack 応答メッセージ

#### 好みが1件以上抽出できた場合
```
✅ フィードバックを記録しました。次回から以下の好みを反映します：
• Terraformコードはmoduleを分割してディレクトリ構造で管理する
（現在 N 件の好みが登録されています）
```

#### 好みが抽出できなかった場合（preferences が空）
```
📝 フィードバックを受け取りました。具体的な好みを抽出できませんでしたが、参考にします。
もう少し詳しく教えていただけると、より精度の高い反映ができます。
```

#### エラー発生時
```
❌ フィードバックの反映中にエラーが発生しました。ご不便をおかけします。
```

### AC-F8-7: 生成への反映（orchestrator.py）

- [ ] `run()` 関数内でユーザープロファイルを取得した後、`learned_preferences` の有無を確認する
- [ ] `learned_preferences` が1件以上ある場合、以下の形式でトピック解析プロンプトおよびワークフロー設計プロンプトに追記する：
  ```
  ## ユーザーの蓄積された好み（学習済み）
  以下の好みを成果物の生成方針に必ず反映してください：
  - Terraformコードはmoduleを分割してディレクトリ構造で管理する
  - 説明は箇条書きで要点のみ、本文を長くしない
  ```
- [ ] `learned_preferences` が空の場合はプロンプトへの追記を行わない（既存動作に影響しない）

## 4. データモデル変更

### user-profiles テーブル（変更）

既存フィールドはそのまま。`learned_preferences` フィールドを追加する。
DynamoDB はスキーマレスのため、既存レコードへの影響なし（フィールド未存在は空リストとして扱う）。

```
USER_PROFILE（変更後）
{
  user_id: string (PK)              ← 変更なし
  role: string                      ← 変更なし
  clouds: list<string>              ← 変更なし
  tech_stack: list<string>          ← 変更なし
  expertise: list<string>           ← 変更なし
  interests: list<string>           ← 変更なし
  org_context: string               ← 変更なし
  learned_preferences: list<object> ← 【新規追加】最大10件
    └── {
          text: string        好みを表す命令形の文章
          created_at: string  ISO 8601 UTC
        }
  created_at: string                ← 変更なし
  updated_at: string                ← 変更なし（フィードバック処理時も更新）
}
```

### workflow-executions テーブル（変更なし）

既存の `slack_thread_ts` フィールドをフィードバック検出のキーとして流用するため、スキーマ変更は不要。

## 5. エラーケースと対処方針

| エラーケース | 対処 |
|------------|------|
| Lambda での DynamoDB スキャンがタイムアウト | フィードバック検出を中止し、新規トピックとしてフォールバック処理 |
| ECS タスク起動失敗 | Lambda は 200 を返した後のため、Slack への通知なし。CloudWatch Logs にエラーを記録 |
| 実行レコードが見つからない（ECS 内） | `topic="不明"`, `category="不明"` でフィードバック解析を続行 |
| Claude の JSON パースに失敗 | `preferences = []` として扱い、「抽出できなかった」メッセージを Slack に投稿 |
| `learned_preferences` の更新で DynamoDB エラー | エラーメッセージを Slack に投稿し、プロファイルは更新されない状態で終了 |
| フィードバックテキストが空文字（絵文字のみ等） | Claude に渡してそのまま解析（`preferences = []` になる可能性が高い） |

---

## 6. 非機能要件

| 項目 | 要件 |
|------|------|
| Lambda ACK 応答時間 | フィードバック検出から 3 秒以内に Slack へ ACK を投稿 |
| フィードバック反映完了時間 | ACK から 60 秒以内に完了通知を Slack に投稿 |
| プロファイル更新の原子性 | `learned_preferences` の上書きは1回の DynamoDB `put_item` で完了（部分更新なし） |
| 既存機能への影響 | `learned_preferences` が空の既存ユーザーに対して既存フローに変更なし |

---

## 7. 制約事項

### 機能制約
- フィードバックは完了通知スレッド内でのみ受け付ける（別スレッドや新規 DM は新規トピックとして処理される）
- `learned_preferences` はユーザー単位の汎用的な好みとして管理する（トピック・カテゴリ別の好み管理はしない）
- フィードバックを複数回送った場合、それぞれ独立して処理・蓄積される（最大10件に達すると古いものが削除）

### アーキテクチャ制約
- 新規 AWS リソース（Lambda, DynamoDB テーブル, SQS 等）は追加しない
- 既存の ECS タスク起動フローと同じネットワーク設定・ロールを流用する
- Lambda のタイムアウト設定（10秒）は変更しない

### スコープ外（明示的に対象外）
- フィードバックに基づく**既存成果物の再生成**
- 「フィードバックを取り消す」「好みをリセットする」操作
- カテゴリ別・トピック別の好み管理
- フィードバックの履歴閲覧（直近の更新内容は更新時の Slack 通知で確認）
- Slack リアクション（👍/👎）によるフィードバック
- フィードバック内容の編集・削除
