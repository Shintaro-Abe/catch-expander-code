# 学習済み好みの関連性判定は抽出時 1 回の分類 + 実行時の決定的フィルタで行う

F8 フィードバック学習の `learned_preferences` は導入以来フラットな `{text, created_at}` リストで、全件が
全プロンプト（トピック解析・ワークフロー設計・生成）に「必ず反映してください」で注入されていた。この結果、
コード成果物由来の好み（例:「module分割する」）が時事レポートの生成にも混入し、成果物品質を下げる障害が
発生した。「どの好みがどのトピック・成果物に関連するか」の判定を**どこに置くか**が本 ADR の対象。

採用: 好みに適用スコープ（トピックカテゴリ × 成果物区分の 2 次元、既存 enum に閉じる）を**抽出時に 1 回だけ
LLM が付与**して保存し、実行時はコードが `preference_applies()` で**決定的にフィルタ**する。パイプラインは
段階的絞り込み（①解析 = 汎用のみ / ②設計 = 汎用 + カテゴリ一致 / ③生成 = 完全フィルタ）を取る。

## 前提となる事実

- カテゴリ 5 値（orchestrator.md のトピック解析出力）と成果物タイプ 7 種は既存の閉じた語彙で、判定材料は
  実行レコード（category / deliverable_types）としてすでに永続化されている
- 好みは単一ユーザー・最大 20 件の規模。埋め込み検索等の類似度基盤は過剰
- 好みスコープの成果物区分は 6 値で、「コード」⊃ {iac_code, program_code} の展開はコード側の固定マップ
  （ワークフロー内部の 7 種 enum は不変。ユーザーの認識では IaC / プログラムの区別は「コード」で一つ）

## Considered Options

- **プロンプト文言の緩和のみ**（「関連する好みのみ反映」に変更）— 変更 1 行だが判定が毎回 LLM 任せで、
  ミスマッチ再発の余地が残る症状パッチのため主軸としては却下（③の文言使い分けとしてのみ併用）
- **実行時 LLM 判定**（トピック解析に「適用すべき好みインデックス選択」を追加出力させる）— スキーマ変更
  不要だが、実行のたびに判定が揺れて再現性がなく、ユーザーが判定結果を事前に確認・訂正できないため却下
- **自由記述の適用条件**（`applies_when: "IaCコードを生成するとき"`）— 表現力最大だが判定が結局実行時
  LLM 頼みになり、型層に置く意味が失われるため却下

## 判断の核

抽出時 1 回の分類は (1) 判定が実行間で揺れない、(2) スコープが Slack 通知と /profile 画面に**見える**ため
誤分類をユーザーがその場で訂正できる（訂正チャネルは F8 フィードバック自体に一本化）、(3) フィルタが純粋
関数になりユニットテストで障害の再現ケースを固定できる。分類を間違えるリスクは非対称フォールバック
（LLM の意図的な空 = 汎用 / enum 検証失敗 = 元実行のスコープに縮退）で「過剰適用より過少適用」に倒す —
効かなかった好みは再フィードバックで直せるが、混入した好みは成果物品質を直接下げるため。

## Consequences

- `learned_preferences` に `scope` フィールドが追加される。欠損は汎用扱い（防御的読み取り）で後方互換
- 既存データは使い捨てスクリプト（dry-run → 目視確認 → --apply）で一括再分類する
- ② ワークフロー設計には成果物スコープ付き好みが「該当成果物を選ぶ場合に考慮」の緩和文言で提示される
  （成果物タイプ選択の**前**に確定フィルタできない鶏卵問題への対応。生成内容への強制ではない）
- プロファイル JSON（profile_text_base）から learned_preferences を除外する。含めるとフィルタが素通りする
- 保存上限は 10 → 20 件（注入量はフィルタで制御されるため、保存を絞る理由が弱くなった）
