# プロダクト要求定義書

## 1. プロダクトビジョンと目的

### プロダクト名
**Catch Expander**

### ビジョン
ユーザーの「気になる」をキャッチし、AIエージェントが自律的に思考・調査・構成して、**そのトピックに最適な成果物**へ拡張（Expand）する。

従来の検索やチャットAIとの違いは、**ユーザーが「何を調べるか」「何を作るか」を指示しない**ことにある。ユーザーはトピックを投げるだけで、AIエージェントが「このトピックなら何を調べ、どんな成果物を作れば最も価値があるか」を自ら判断し、ワークフローを組み立てて実行する。

成果物はテキストレポートに限らない。トピックに応じて、IaCコード、設計書、プログラムコード、比較表、見積もりなど、**ユーザーがそのまま業務に使える形式**でアウトプットする。

### コアコンセプト: Catch → Expand

```
[ユーザー]                    [Catch Expander]
    |                              |
    |  "AIパイプライン"            |
    |----------------------------->|
    |                              | 1. トピックを解析
    |                              | 2. ユーザープロファイルを参照
    |                              | 3. 最適なワークフローを自律設計
    |                              |    - 概要調査が必要 → 実行
    |                              |    - 論文調査が必要 → 実行
    |                              |    - クラウド実装例が有用 → 実行
    |                              |    - 料金見積もりが有用 → 実行
    |                              |    - IaCコードが有用 → 生成
    |                              |    - 設計書が有用 → 作成
    |                              | 4. 調査結果と成果物を構成
    |  <レポート + 成果物>         |
    |<-----------------------------|
```

核心は**ステップ3**にある。エージェントはトピックのカテゴリやユーザーの専門領域に応じて、毎回異なるワークフローを組み立てる。テンプレートの穴埋めではなく、トピックごとに「何を調べるべきか」「どんな成果物を作るべきか」「どう構成すべきか」を思考する。

調査だけでなく**成果物の生成もワークフローに含まれる**。技術トピックならIaCコードや設計書、ビジネストピックなら比較表や提案書など、エージェントがトピックに応じて必要な成果物の種類と形式を自律的に判断する。

### 目的
- ユーザーの知的好奇心や業務上の関心を、**即座にアクショナブルな成果物**へ変換する
- 情報収集・整理・成果物作成の全工程をAIエージェントが自律的に設計・実行する
- ユーザーの背景情報を活かし、そのユーザーが**そのまま業務に使える形式**で出力する

## 2. ターゲットユーザーと課題・ニーズ

### ターゲットユーザー
**プライマリ：マルチクラウド対応のクラウドエンジニア**

- AWS、Azure、Google Cloud等の複数クラウドにまたがるインフラの設計・構築・運用を担当
- 新技術のキャッチアップを日常的に行う
- 技術選定や提案資料の作成が業務に含まれる
- クラウドベンダ横断での比較・最適解の判断が求められる

**セカンダリ：技術以外の情報も素早く把握したい技術者**

- 国際情勢やビジネストレンドも業務判断の材料にする
- 調査に時間を割けないが、質の高いインプットを求める

### 課題

| 課題 | 具体例 |
|------|--------|
| **調査の初動が重い** | 「AIパイプライン」が気になっても、何から調べるか決めるだけで時間がかかる |
| **横断的調査のコスト** | 概要→論文→実装例→料金と複数観点を調べるには数時間〜数日かかる |
| **コンテキストの欠落** | 一般的な調査結果では、自分の技術スタックや担当クラウドに合った情報が埋もれる |
| **構成・整理の手間** | 調べた内容を提案として使える形にまとめる作業が別途必要 |
| **成果物作成の手間** | 調査後にIaCコードや設計書を自分で書く必要がある |

### ニーズ
- キーワードを1つ送るだけで、**自分のコンテキストに最適化された**調査結果と成果物が欲しい
- 「何を調べるか」「何を作るか」から考えてくれるエージェントが欲しい
- 調査レポートだけでなく、コードや設計書など**すぐに使える成果物**まで出して欲しい
- Slackという日常ツールで完結したい

## 4. 成功の定義

### 定量指標

| 指標 | 目標値 |
|------|--------|
| 提案完了率 | トピック送信の90%以上が提案返信まで完了 |
| 初回応答時間 | 処理開始通知がSlackに届くまで3秒以内 |
| 提案生成時間 | トピック送信から提案返信まで5分以内 |
| ユーザー再利用率 | 初回利用から1週間以内に再度トピックを送信する割合が50%以上 |

### 定性指標
- 成果物が「追加調査や作成作業なしで業務に使える」とユーザーが判断する
- ワークフローの構成がトピックに対して的確（技術トピックならコード・設計書を含む、時事トピックならレポートのみ等）
- ユーザープロファイルの反映が成果物に明確に表れる（例：担当クラウドに合わせたIaCコード）
- コード成果物がそのまま動作する、または最小限の修正で使える品質である

## 5. ビジネス要件

- **利用規模**：個人〜小規模チーム（初期は1ワークスペース）
- **コスト**：月間運用コストを予測可能な範囲に抑える（サーバーレス従量課金）
- **外部API依存**：Claude Opus/Maxプラン＋Claude Code CLI（成果物生成）、Notion API（成果物格納）、GitHub API（コード格納）
- **備考**：Web検索はClaude Code CLI組み込みのWebSearch/WebFetchを利用（外部API別途契約不要）
- **運用**：デプロイ後は基本的にメンテナンスフリーで稼働

## 6. ユーザーストーリー

### US-1: トピック送信と成果物受信
**As a** クラウドエンジニア  
**I want to** Slackで「AIパイプライン」とだけ送信する  
**So that** 概要レポート、クラウド別実装手順、料金見積もり、さらにIaCコードや設計書まで含めた成果物一式が返ってくる

### US-2: ユーザーコンテキストの反映
**As a** マルチクラウドを担当するエンジニア  
**I want to** 自分のロール・技術スタック・担当クラウド・関心領域を事前に登録しておく  
**So that** 提案に自分の担当クラウド（AWS、Azure、Google Cloud等）に応じた実装手順や料金情報が自動的に含まれる

### US-3: 技術以外のトピック
**As a** ユーザー  
**I want to** 「イラン戦争」のような時事トピックも送信できる  
**So that** 背景、ニュースダイジェスト、今後の予測など、そのトピックに適した切り口で提案が返る

### US-4: 処理中の進捗把握
**As a** ユーザー  
**I want to** エージェントが何を調査中か進捗を知りたい  
**So that** 待ち時間に不安を感じず、提案の信頼性を事前に把握できる

### US-5: 成果物の深掘り
**As a** ユーザー  
**I want to** 返ってきた成果物の特定セクションについて追加質問や修正依頼をしたい  
**So that** レポートの深掘りやコードの調整ができる

### US-6: 成果物としてのコード取得
**As a** クラウドエンジニア  
**I want to** 技術トピックに対してIaCコードやプログラムコードも成果物として受け取りたい  
**So that** 調査結果をすぐに実装に移せる（コードをコピーしてそのまま使い始められる）

### US-7: フィードバックの送信（F8）
**As a** ユーザー  
**I want to** 成果物の完了通知スレッドで感想・改善要望を自由形式で書く  
**So that** 特別な操作や形式を覚えることなく自分の好みをシステムに伝えられる

### US-8: フィードバックの反映確認（F8）
**As a** ユーザー  
**I want to** フィードバックが受け取られ、どんな好みとして記録されたかを同じスレッドで知りたい  
**So that** 記録内容が意図と一致しているか確認でき、誤解があればすぐに追加フィードバックで修正できる

### US-9: 累積的な品質改善（F8）
**As a** ユーザー  
**I want to** 複数回フィードバックを繰り返すことで、好みのリストが充実していく  
**So that** 使い込むほど自分向けにカスタマイズされた成果物が得られる

### US-10: 過去成果物の履歴確認（F9）
**As a** Catch-Expander の利用者として、  
**I want** Slack 上で簡単に過去の成果物リストを確認したい、  
**So that** 既存の調査結果を再利用したり、別のチャンネルで共有したりできる。

**受け入れ条件**: AC-7 を参照

## 7. 受け入れ条件

### AC-1: 基本フロー（トピック→成果物）
- [ ] Slackでメッセージ（メンションまたはDM）を送信すると、3秒以内に処理開始通知が返る
- [ ] エージェントがトピックのカテゴリ・深掘り観点・必要な成果物タイプを自律的に判断する
- [ ] 判断に基づいてワークフロー（調査＋成果物作成）を動的に構築し、実行する
- [ ] 成果物一式が格納先に保存される
- [ ] Slackに成果物サマリーと格納先リンクが通知される
- [ ] 成果物の種類と構成がトピックのカテゴリに応じて異なる

### AC-2: 成果物の適切性
- [ ] 技術トピック（例：「AIパイプライン」）の場合：
  - 概要説明が含まれる
  - 関連する論文・技術リソースへの参照が含まれる
  - 具体的なユースケースが含まれる
  - ユーザーの担当クラウドに合わせた実装手順が含まれる
  - 料金見積もり・コスト情報が含まれる
  - **IaCコード（Terraform等）が成果物として含まれる**
  - **アーキテクチャ設計書が成果物として含まれる**
- [ ] 時事トピック（例：「イラン戦争」）の場合：
  - 事象の背景・経緯が含まれる
  - 最新ニュースのダイジェストが含まれる
  - 今後予測されるシナリオが含まれる
- [ ] エージェントがトピックに応じて「どの種類の成果物が必要か」を自律的に判断する
- [ ] 成果物にコードが含まれる場合、格納先から直接クローン/ダウンロードして使える形式で提供される

### AC-3: ユーザープロファイル
- [ ] ユーザーのプロファイル情報（ロール、技術スタック、関心領域）を登録できる
- [ ] 提案生成時にプロファイルが参照され、内容に反映される
- [ ] 担当クラウドが複数の場合、ベンダ横断の比較情報が含まれる
- [ ] 同じトピックでもプロファイルが異なれば、異なる切り口の提案が生成される

### AC-4: 進捗通知
- [ ] 処理開始時に「リサーチを開始します」等の通知がSlackに送られる
- [ ] ワークフローの主要ステップごとに進捗が通知される（例：「論文を検索中...」）
- [ ] エラー発生時はユーザーに通知され、部分的な結果がある場合はそれを返す

### AC-5: 品質担保
- [ ] 成果物内のすべての事実主張に出典URLが記載されている
- [ ] 出典URLが実在し、記述内容と整合していることが検証されている
- [ ] 出典はカテゴリ別のソース優先順位に従って選択されている（公式情報が最優先）
- [ ] 検証できない主張には「未検証」マークが付与されている
- [ ] カテゴリ別のセルフレビューチェックリストを全項目合格している
- [ ] 成果物末尾に品質メタデータ（検証ステータス、情報鮮度、レビュー結果）が記載されている

### AC-6: フィードバック学習（F8）
- [ ] 完了通知スレッドへの自由テキスト投稿が自動的にフィードバックとして検出される
- [ ] フィードバック受信後3秒以内に同スレッドへACKメッセージが投稿される
- [ ] Claude がフィードバックから具体的な好みを抽出し、命令形テキストとして `learned_preferences` に保存される
- [ ] 好みが抽出できた場合、記録した内容の一覧と総件数が同スレッドに通知される
- [ ] 具体的な好みが抽出できない場合も「参考にします」旨が通知される
- [ ] 複数回フィードバックを繰り返すと `learned_preferences` が累積される（最大10件）
- [ ] 次回のトピック処理時に `learned_preferences` がプロンプトに反映され、成果物に表れる
- [ ] フィードバックはユーザーが手動登録したプロファイル項目（`role`, `clouds` 等）を上書きしない
- [ ] 処理中の実行（`status != "completed"`）のスレッドへの返信は無視される

### AC-7: 成果物履歴コマンド（F9）

- [ ] Slack で `履歴` または `history [keyword]` を投稿すると、直近の完了成果物が一覧表示される
- [ ] 表示件数は最大 5 件
- [ ] 各エントリには Notion URL (📝) が表示される
- [ ] コード成果物がある場合は GitHub URL (💻) も表示される
- [ ] keyword 指定時はトピックに対する部分一致で絞り込まれる
- [ ] 過去レコード（フィールド `github_url` 不在の旧レコード）でもエラーなく動作する

## 8. 機能要件

### FR-1: Slack連携

#### 入力
- Slack Bot Appとしてワークスペースにインストール
- 以下の方法でトピックを受け付ける：
  - Bot宛てのDM
  - チャンネルでのメンション（`@CatchExpander AIパイプライン`）
- 入力は自由テキスト（キーワード、短文、質問形式いずれも可）

#### 出力（Slackは通知のみ）
Slackは**進捗通知と完了通知の経路**として使用する。成果物本体はSlackには投稿せず、外部の格納先に一式を保存し、Slackからリンクで参照する。

- 完了通知に成果物一覧（ファイル名・概要）とアクセスリンク（Notion/GitHub）を含める
- 成果物のサマリー（要点の要約）をSlackメッセージに含め、詳細は格納先で閲覧

#### 成果物の格納先

成果物の分野は技術・時事・ビジネスなど多岐にわたるため、格納先は**コンテンツの性質に応じた2系統**で構成する。エージェントがトピック解析時にコード成果物の有無を判断し、格納先を自動選択する。

**Notion（全トピック共通・主格納先）**
- 全トピックの成果物をNotionのDatabase内にページとして格納
- レポート、設計書（Mermaid図）、比較表、見積もりはNotionに集約
- トピック名・カテゴリ・日付でフィルタ/ソート可能
- Catch Expander専用のIntegrationを作成し、成果物DBのみに共有（他ページへのアクセス不可）

**GitHub Repository（コード成果物がある場合のみ・補助格納先）**
- IaCコード、プログラムコード等のコード成果物をリポジトリに格納
- `git clone` で取得後すぐに実行可能な状態（ファイル構成・変数定義・使用手順を含む）
- NotionページからGitHubへ、GitHubのREADMEからNotionへ双方向リンク
- コード成果物がないトピック（時事、ビジネス等）ではGitHubへの出力は行わない

**エージェントの格納先選択ロジック**
```
トピック解析 → 成果物タイプ判断
  ├── コード成果物あり → Notion（レポート類）+ GitHub（コード類）
  └── コード成果物なし → Notionのみ
```

**Notion Integrationの安全設計**
- Integrationは成果物DBのみに共有し、ワークスペース内の他ページにはアクセス不可
- 他のワークスペースへのアクセスはNotion側のアーキテクチャで不可能
- エージェントのアプリケーショ��層で以下を制御：
  - 自身が作成したページのみ更新可能（推敲のため更新APIは使用する）
  - 完了済みステータスのページは更新しない
  - ページ削除APIは一切呼ばない

#### 進捗通知
- 処理開始時：「トピックを受け取りました。リサーチを開始します。」
- ワークフロー構築後：「以下の計画で進めます：調査[1.○○ 2.○○] → 成果物作成[設計書, Terraformコード]」
- 調査ステップ完了時：「○○の調査が完了しました（2/4）」
- 成果物生成時：「Terraformコードを生成中...」
- 完了時：成果物サマリー＋格納先リンク（Notion、該当時はGitHubも）を通知

### FR-2: トピック解析エンジン

入力テキストを解析し、以下を判定する：

| 判定項目 | 説明 | 例 |
|----------|------|-----|
| カテゴリ | トピックの分野 | 技術、時事、ビジネス、学術、カルチャー等 |
| 意図 | ユーザーが知りたいことの方向性 | 技術導入検討、情報キャッチアップ、意思決定支援等 |
| 深掘り観点 | 調査すべき切り口のリスト | 概要、実装方法、コスト、リスク、最新動向等 |

解析結果はユーザープロファイルと組み合わせて、FR-3のワークフロー構築に渡される。

### FR-3: 自律ワークフロー構築エンジン

**本プロダクトの中核機能。**

エージェントはトピック解析結果とユーザープロファイルを入力として、以下を自律的に決定する：

1. **調査項目の決定** — このトピックで何を調べるべきか
2. **成果物タイプの決定** — どんな種類の成果物を作るべきか（レポート、コード、設計書等）
3. **実行順序の決定** — どの順序で調査・生成すると効率的か
4. **構成の決定** — 成果物全体をどう構成するか

#### 成果物タイプの例

エージェントはトピックとユーザープロファイルに基づき、以下の中から必要なものを自律的に選択・組み合わせる：

| 成果物タイプ | 説明 | 適用例 |
|-------------|------|--------|
| 調査レポート | 概要、背景、分析等のテキスト | 全トピック共通 |
| IaCコード | Terraform、CloudFormation等 | クラウドインフラ系トピック |
| プログラムコード | Python、TypeScript等のアプリコード | 技術実装系トピック |
| アーキテクチャ設計書 | 構成図、データフロー等 | システム設計系トピック |
| 比較表 | ベンダ比較、技術比較 | 技術選定系トピック |
| 料金見積もり | クラウドサービスのコスト試算 | クラウド系トピック |
| 手順書 | ステップバイステップの実装/運用手順 | 導入・構築系トピック |

成果物タイプは固定リストではなく、エージェントがトピックに応じて柔軟に判断する。上記はあくまで典型例である。

#### ワークフロー設計の例

**例1：技術トピック「AIパイプライン」× クラウドエンジニア（担当：AWS, Google Cloud）**
```
ワークフロー:
  [調査フェーズ]
  1. [概要調査]     AIパイプラインの定義・主要コンポーネントを検索
  2. [論文・事例]   最新の論文・技術ブログを検索
  3. [ユースケース]  業界別の導入事例を検索
  4. [クラウド実装]  担当クラウドでの構築手順を検索
                    - AWS: SageMaker, Step Functions等
                    - Google Cloud: Vertex AI Pipelines等
  5. [コスト試算]   各クラウドサービスの料金情報を収集し見積もり・比較

  [成果物生成フェーズ]
  6. [設計書作成]   アーキテクチャ設計書（構成図・データフロー）を作成
  7. [IaCコード生成] Terraformコード（AWS/GCP両対応）を生成
  8. [レポート構成]  調査結果＋成果物を統合して最終レポートを構成

  [品質担保フェーズ]
  9.  [ソース検証]   全出典URLの実在確認＋内容整合性チェック
  10. [セルフレビュー] カテゴリ別チェックリストで自己評価・修正
  11. [品質メタデータ] 検証結果・鮮度・注意事項を付与

成果物一式:
  - 調査レポート（概要、論文、ユースケース、クラウド別実装手順、料金比較）
  - アーキテクチャ設計書（Mermaid図）
  - Terraformコード（AWS版 / Google Cloud版）
  - まとめと推奨アクション
  - 品質メタデータ（出典検証結果、セルフレビュー結果）
```

**例2：時事トピック「イラン戦争」× クラウドエンジニア**
```
ワークフロー:
  [調査フェーズ]
  1. [背景調査]     紛争の歴史的背景・関係国を検索
  2. [最新情報]     直近のニュース・報道を検索
  3. [分析]         専門家の分析・見解を検索
  4. [影響予測]     経済・地政学的影響の予測を検索

  [成果物生成フェーズ]
  5. [レポート構成]  調査結果を時系列・論点別に構成
  ※ コード・設計書は不要とエージェントが判断 → 生成しない

  [品質担保フェーズ]
  6. [ソース検証]   全出典URLの実在確認＋内容整合性チェック
  7. [セルフレビュー] 時事カテゴリのチェックリストで自己評価・修正
  8. [品質メタデータ] 検証結果・鮮度・注意事項を付与

成果物一式:
  - 調査レポート（背景、ニュースダイジェスト、専門家見解、予測シナリオ）
  - まとめ
  - 品質メタデータ（出典検証結果、セルフレビュー結果）
```

ワークフローのポイント：
- **調査フェーズと成果物生成フェーズ**の2段階で構成される
- 成果物生成フェーズで何を作るかはエージェントが自律的に判断する
- 時事トピックのようにコードが不要な場合、エージェントは成果物生成をスキップする
- 同じ「AIパイプライン」でも、ユーザーがデータサイエンティストであれば「IaCコード」の代わりに「Jupyterノートブックのサンプルコード」を生成するなど、プロファイルによって成果物の種類が変わる
- 担当クラウドがAWSのみであればAWS特化、複数担当であればベンダ比較を含む構成になる

### FR-4: 情報収集

- Web検索APIを利用して各調査項目の情報を収集
- 検索クエリはワークフローの各ステップに応じてエージェントが生成
- 取得した情報はエージェントが要約・取捨選択し、提案に組み込む
- 情報ソースのURL・タイトルを提案内に出典として記載

### FR-5: 成果物生成

エージェントがワークフローで決定した成果物タイプに応じて、以下を生成する：

#### テキスト成果物（調査レポート、比較表、手順書等）
- ワークフローで決定した構成に従い、収集情報を配置
- 各セクションの分量はトピックの重要度に応じてエージェントが調整
- ユーザープロファイルのコンテキストを反映した表現・切り口を使用
- 末尾に「まとめと推奨アクション」セクションを含め、次のステップを提示

#### コード成果物（IaCコード、プログラムコード等）
- ユーザーの技術スタックに合わせた言語・フレームワークで生成
- コピー＆ペーストで使える形式（コードブロック）で出力
- コード内にコメントで説明を付与
- 必要に応じてディレクトリ構成や使用方法の説明を添付

#### 設計書成果物（アーキテクチャ設計書等）
- Mermaid記法で構成図・データフロー図を生成
- 設計判断の根拠を付記
- ユーザーの担当クラウドに合わせたサービス構成を記載

#### 成果物の保存と通知

エージェントがコード成果物の有無を判断し、格納先を自動選択する。

**コード成果物ありの場合（例：AIパイプライン）**
```
Notion（成果物DB内のページ）:
  [AIパイプライン] 2026-04-04  カテゴリ:技術
  ├── 調査レポート（ページ内セクション）
  ├── アーキテクチャ設計書（Mermaid図埋め込み）
  ├── 料金見積もり比較（テーブル）
  └── 🔗 コードはこちら → GitHub リンク

GitHub（catch-expander-code リポジトリ）:
  ai-pipeline-20260404/
  ├── README.md（概要 + Notionページへのリンク）
  ├── aws/
  │   ├── main.tf
  │   ├── variables.tf
  │   └── outputs.tf
  └── gcp/
      ├── main.tf
      ├── variables.tf
      └── outputs.tf
```

**コード成果物なしの場合（例：イラン戦争）**
```
Notion（成果物DB内のページ）:
  [イラン戦争] 2026-04-05  カテゴリ:時事
  ├── 背景と経緯
  ├── ニュースダイジェスト
  └── 今後の予測シナリオ

GitHub: 出力なし
```

- Slackには成果物一覧・サマリー・格納先リンク（Notion、該当時はGitHubも）のみ通知

### FR-6: ユーザープロファイル管理

#### プロファイル項目

| 項目 | 説明 | 例 |
|------|------|-----|
| ロール | 職種・役割 | クラウドエンジニア |
| 担当クラウド | 業務で使用するクラウドベンダ | AWS, Google Cloud |
| 技術スタック | 主に使用する技術 | Terraform, Python, Kubernetes |
| 専門領域 | 深い知識を持つ分野 | インフラ設計、CI/CD |
| 関心領域 | 現在関心のあるテーマ | サーバーレス、AI/ML |
| 組織コンテキスト | 業務上の背景 | SaaS企業のインフラチーム所属 |

#### 登録方法
- Slackコマンド（`@CatchExpander profile`）でプロファイル登録・更新
- プロファイルはJSON形式で保存

#### 提案への影響
- 担当クラウドが「AWS, Google Cloud」→ 両ベンダでの実装手順・比較を含める
- 担当クラウドが「AWS」のみ → AWS特化の提案にする
- ロールが「エンジニア」→ 技術的な詳細を含める
- 関心領域に「サーバーレス」→ サーバーレス関連の情報を優先的に収集

### FR-7: 品質担保

成果物の信頼性を確保するため、全カテゴリの成果物に対して3層の品質担保を実施する。

#### 第1層: ソース検証

成果物に記載するすべての事実主張に出典URLを紐づけ、出典の実在と内容の整合性を検証する。

**ソース優先順位（技術トピック）**

| 優先度 | ソース種別 | 例 |
|:---:|------------|-----|
| 1 | 公式ドキュメント | AWS公式ドキュメント、Terraform Registry、Google Cloud公式ドキュメント |
| 2 | 公式ブログ・発表 | AWS Blog、Google Cloud Blog、公式リリースノート |
| 3 | 査読済み論文 | arXiv（査読済み）、IEEE、ACM |
| 4 | 著名技術ブログ・カンファレンス資料 | re:Invent発表資料、Google I/Oセッション |
| 5 | 技術コミュニティ | Stack Overflow（高スコア回答）、Zenn、Qiita（検証可能な内容に限る） |
| 6 | 個人ブログ・SNS | 補足情報としてのみ使用。事実の根拠にはしない |

**ソース優先順位（時事トピック）**

| 優先度 | ソース種別 | 例 |
|:---:|------------|-----|
| 1 | 通信社・公的機関 | Reuters、AP、AFP、各国政府の公式発表 |
| 2 | 大手報道機関 | BBC、NHK、日経、NYT、WSJ |
| 3 | 専門メディア | 分野特化の報道機関（軍事、経済等） |
| 4 | シンクタンク・研究機関 | CSIS、Brookings、RIETI等の分析レポート |
| 5 | 専門家の解説 | 著名な研究者・ジャーナリストの署名記事 |
| 6 | SNS・個人発信 | 当事者の一次発言としてのみ使用。分析の根拠にはしない |

**ソース優先順位（ビジネス・トレンドトピック）**

| 優先度 | ソース種別 | 例 |
|:---:|------------|-----|
| 1 | 公的統計・政府データ | 総務省統計局、BLS、World Bank |
| 2 | 企業の公式IR・決算 | 年次報告書、決算発表、プレスリリース |
| 3 | 大手調査会社 | Gartner、IDC、Forrester |
| 4 | 業界団体・規制当局 | 各業界協会の発表、金融庁等の報告 |
| 5 | ビジネスメディア | Harvard Business Review、日経ビジネス |
| 6 | 個人の見解・予測 | 補足としてのみ使用 |

**ソース運用ルール**

| ルール | 内容 |
|--------|------|
| 上位ソース優先 | 同じ事実について上位と下位のソースが矛盾する場合、上位ソースを採用 |
| 複数ソース推奨 | 重要な事実主張は2つ以上のソースで裏付ける |
| 鮮度の考慮 | 同一優先度のソースでは新しい方を優先 |
| 下位ソースの活用制限 | 優先度5〜6のソースは補足・参考としてのみ使用し、事実の根拠にはしない |
| ソース不在時の扱い | 信頼できるソースが見つからない主張は「未検証」として明示 |
| 料金情報 | 公式料金ページのみを使用。ブログ記事の料金情報は採用しない |
| 出典の記載形式 | 成果物内の各事実主張に出典URLを記載する |

**検証チェック項目**

| チェック項目 | 内容 |
|-------------|------|
| 出典の実在性 | URLにアクセスして実在することを確認 |
| 出典との整合性 | 出典の内容と成果物の記述が矛盾しないことを確認 |
| 出典の鮮度 | 出典の公開日が古すぎないことを確認（時事トピックは特に重要） |
| 出典のないファクト | 出典を付けられない事実主張に「未検証」を明示 |

#### 第2層: カテゴリ別セルフレビュー

成果物生成とは**別のLLM呼び出し**で、カテゴリに応じたチェックリストを用いてレビューする。生成時のコンテキストをリセットすることで、自身の出力を正しいと思い込むバイアスを軽減する。

**レビューフロー**
```
LLM呼び出し①: 成果物生成（下書き→推敲）
    ↓
LLM呼び出し②: セルフレビュー
  入力: 成果物 + カテゴリ別チェックリスト + 出典URL一覧
  出力: 各項目の合否判定 + 不合格項目の修正指示
    ↓
  合格 → 第3層へ
  不合格 → LLM呼び出し③: 修正 → LLM呼び出し④: 再レビュー
    ↓
  修正ループ上限: 最大2回（実装しながら調整）
  上限到達時: 残りの指摘事項を品質メタデータの注意事項に記載して成果物を出力
```

成果物を出さないよりも、注意付きで出す方がユーザーにとって有用であるため、上限到達時も成果物は出力する。

**技術トピック（レポート・設計書）**

| # | チェック項目 |
|---|-------------|
| 1 | 記載されたサービス名・機能名は現在も存在するか |
| 2 | 非推奨・廃止されたサービスを推奨していないか |
| 3 | 設計書の構成図とレポートの実装手順に矛盾がないか |
| 4 | 料金情報の出典は公式料金ページか |

**技術トピック（コード）**

| # | チェック項目 |
|---|-------------|
| 1 | コードの構文が正しいか（可能な場合はLintを実行） |
| 2 | 使用しているAPIバージョン/プロバイダバージョンは最新か |
| 3 | ハードコードされたシークレットが含まれていないか |
| 4 | READMEの使用手順通りに実行可能な構成になっているか |

**時事トピック**

| # | チェック項目 |
|---|-------------|
| 1 | 日付・時系列に誤りがないか |
| 2 | 人物名・国名・組織名が正確か |
| 3 | 複数の視点（当事者、第三者、専門家）が含まれているか |
| 4 | 事実と分析/予測が明確に区別されているか |
| 5 | 出典が特定の立場に偏っていないか |

**ビジネス・トレンドトピック**

| # | チェック項目 |
|---|-------------|
| 1 | 統計データの出典と時期が明記されているか |
| 2 | 因果関係と相関関係を混同していないか |
| 3 | 予測やトレンドの根拠が示されているか |

#### 第3層: 品質メタデータの付与

各成果物の末尾に品質情報を記載し、ユーザーが信頼度を判断できるようにする。

```
■ 品質情報

検証ステータス:
  ✅ 出典検証済み: 12件
  ⚠️ 未検証の記述: 2件（セクション3の市場規模、セクション5の予測）

情報の鮮度:
  最新の出典: 2026-04-02
  最古の出典: 2024-11-15

セルフレビュー結果:
  チェック項目: 5/5 合格

注意事項:
  - コード成果物はPoC品質です。本番利用前にレビュー・検証を推奨します
  - 料金見積もりは2026年4月時点の公式料金に基づきます
  - 「未検証」マークの箇所はユーザー自身での確認を推奨します
```

### FR-8: フィードバック学習

#### フィードバックの受付

- 完了通知スレッドへの自由テキスト返信をフィードバックとして自動検出
- DM・チャンネルメンション（`@CatchExpander`）いずれのフィードバックも受け付ける
- 入力形式は自由（「コードが短い」「説明が長い」「次回は○○を含めて」等）

#### フィードバックの解析と保存

- Claude がフィードバックを解析し、次回の成果物生成に反映できる「好み（preference）」を抽出
- 好みは「生成プロンプトにそのまま使える命令形の文章」として保存
- 1回のフィードバックから最大3件の好みを抽出
- 既存の好みと矛盾・重複する場合は既存を置き換え
- `learned_preferences` に最大10件を累積保存（超過時は最古から削除）
- フィードバックは `learned_preferences` フィールドのみを更新し、ユーザーが登録した `role` / `clouds` 等は一切変更しない

#### 生成への反映

- 次回のトピック処理時、`learned_preferences` が1件以上あればオーケストレーターのプロンプトに追記
- トピック解析・ワークフロー設計・成果物生成の全ステップに反映される

## 9. 非機能要件

### NFR-1: パフォーマンス
- 処理開始通知（ACK）：Slackメッセージ受信から3秒以内
- 提案生成完了：トピック送信から5分以内（目標3分）
- Web検索：1ステップあたり10秒以内

### NFR-2: 可用性
- サーバーレスアーキテクチャで構成し、インフラ管理を不要にする
- エージェントの一部ステップが失敗しても、成功したステップの結果で部分提案を返す
- Claude OAuth トークンは EventBridge Schedule (12 時間間隔) でバックグラウンド refresh され、失敗時は Slack 通知される (詳細は `architecture.md` 4. token_monitor 参照)

### NFR-3: セキュリティ
- ユーザープロファイルの暗号化保存
- APIキーはSecrets Manager等で管理（ソースコードに含めない）
- Slack署名検証（`X-Slack-Signature`）による不正リクエスト防止
- 外部に送信する検索クエリにユーザーの個人情報を含めない

### NFR-4: コスト
- サーバーレス（従量課金）で運用
- LLM APIコールはワークフロー設計1回＋各ステップの要約で構成（不要な呼び出しを抑制）
- 月間コスト目安を把握できるようログ/メトリクスを記録

### NFR-5: 拡張性
- 新しいトピックカテゴリへの対応はプロンプト調整で実現（コード変更不要）
- 情報収集ソースの追加（ニュースAPI、論文DB等）が容易なプラグイン構造
- 複数ユーザー対応への拡張が可能な設計

## 10. 制約事項

| 制約 | 内容 | 対策 |
|------|------|------|
| LLMの知識カットオフ | 学習データ以降の最新情報を持たない | Web検索で最新情報を補完 |
| LLMのハルシネーション | 事実と異なる情報を生成する可能性 | 出典URLを必ず記載し、検証可能にする |
| Slackメッセージ上限 | 1メッセージ4,000文字 | 成果物は外部格納、Slackはサマリー通知のみ |
| Web検索の精度 | 検索結果が常に適切とは限らない | 複数クエリで冗長性を確保 |
| リアルタイム性 | 成果物生成に数分かかる | 進捗通知で体験を改善 |
| コード生成の品質 | LLM生成コードはそのまま本番利用できない場合がある | PoCレベルと明示し、レビュー・検証を推奨する旨を付記 |
| コード成果物の閲覧性 | Slackはシンタックスハイライトが限定的 | コードはGitHubで閲覧（Slackにはサマリーのみ） |
| Notion API制限 | 3リクエスト/秒 | 成果物をまとめて構成してから一括投稿する設計にする |

## 11. スコープ定義

### MVP（初回リリース）
